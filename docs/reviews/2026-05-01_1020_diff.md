## Warnings (4)

### W1 — Linear milestone scan on every render
**File**: `src/hud/MissionEventsPanel.tsx:29–35`
**Agents**: Complexity Analyst, DS&A Reviewer, Efficiency Sentinel

`currentIdx` is computed via a backward linear scan through `SORTED_MILESTONES` on every render. Since the array is pre-sorted by `missionElapsedHours`, this is a O(n) scan where O(log n) binary search would suffice. Runs on every render while the panel component is mounted (even when closed), as the index is computed at the component root.

**Suggested fix**: Wrap in `useMemo` with `[elapsedHours]` dependency and implement a binary search:
```ts
const currentIdx = useMemo(() => {
  let lo = 0, hi = SORTED_MILESTONES.length - 1, idx = 0;
  while (lo <= hi) {
    const mid = (lo + hi) >> 1;
    if (elapsedHours >= SORTED_MILESTONES[mid].missionElapsedHours) { idx = mid; lo = mid + 1; }
    else hi = mid - 1;
  }
  return idx;
}, [elapsedHours]);
```

### W2 — O(n) Set allocation in timer cleanup effect
**File**: `src/hud/MissionEventsPanel.tsx:49`
**Agents**: Complexity Analyst, DS&A Reviewer, Efficiency Sentinel

The `useEffect` cleanup (line 49) creates `new Set(alerts.map(...))` on every alerts dependency change to find stale timer IDs. This is correct but allocates a fresh Set per change. For typical alert counts (0–5) this is negligible, but the pattern is wasteful.

**Suggested fix**: Iterate `timers.current` and check `alerts.some()` instead of building a Set, or build the Set once at the top of the effect body and share it between setup and cleanup via closure.

### W3 — Milestone rows have no memoization; all re-render on every time tick
**File**: `src/hud/MissionEventsPanel.tsx:157–211`
**Agents**: Complexity Analyst, Efficiency Sentinel

The milestone list maps `SORTED_MILESTONES` (18 items) and computes `isPast`/`isCurrent` per item on every render. `elapsedHours` changes every second in LIVE/REPLAY mode, so the full list re-renders every second while the panel is open — even for milestones whose state has not changed. Each item also renders a potentially large `<img>` block.

**Suggested fix**: Extract a `MilestoneRow` component wrapped in `React.memo` with a stable equality check on `isPast`, `isCurrent`, and `m.name`. Alternatively, quantize `elapsedHours` to whole hours (sufficient resolution for 18 milestones across 217h).

---

### W4 — Inline style object in alert list recreated every render
**File**: `src/hud/MissionEventsPanel.tsx:131`
**Agents**: Efficiency Sentinel

`style={{ borderLeft: \`3px solid \${SEVERITY_COLORS[alert.severity] || '#888'}\`, paddingLeft: '8px' }}` creates a new style object for every alert item on every render. For the alert list this is low-impact (typically 0–5 items, infrequent renders), but the pattern is avoidable.

**Suggested fix**: Use a Tailwind arbitrary-value class or a CSS variable, or extract a `getAlertBorderStyle` that caches by severity key.
