## Warnings

### W1: No fetch timeout on external API calls

- **File**: `api/chat.ts:134,154,181`
- **Agent**: Efficiency Sentinel
- **Issue**: Three external `fetch` calls (Gemini text, Gemini image, NASA Images) have no `AbortController` or timeout. If any upstream API hangs, the Vercel serverless function remains open consuming resources until platform timeout. With the addition of Search Grounding (which may add latency to Gemini responses), this becomes more relevant under concurrent load.
- **Suggested approach**: Add `AbortSignal.timeout(8000)` (Node 18+) or a manual `AbortController` with `setTimeout` to all three fetch calls.

## Suggestions

### S1: Pre-normalize quick answer questions at module load

- **File**: `src/data/artemis-knowledge.ts:57-66`
- **Agent**: Complexity Analyst, DS&A Reviewer, Efficiency Sentinel
- **Issue**: `findQuickAnswer()` normalizes each `qa.question` (`.toLowerCase().replace()`) on every call. With 12 entries this is negligible, but the normalization is static data that could be pre-computed at module load to avoid repeated string operations.

### S2: Curated video keyword lookup could use Map

- **File**: `api/chat.ts:26-39`
- **Agent**: Complexity Analyst, DS&A Reviewer
- **Issue**: `CURATED_VIDEOS` is scanned with `.find()` + inner `.some()` — O(V*K) per lookup (V=7 videos, K~3 keywords = ~21 comparisons). A flat `Map<keyword, {videoId, title}>` would give O(1) per token. Not worth changing at current scale (7 entries); flag if video catalog grows beyond ~50.

### S3: Message sanitization triple intermediate array

- **File**: `api/chat.ts:241-244`
- **Agent**: Complexity Analyst, DS&A Reviewer
- **Issue**: `.slice(-20).filter().map()` creates 3 intermediate arrays. For max 20 elements the cost is negligible. A single `reduce()` or for-loop would eliminate 2 intermediate allocations but sacrifices readability for no meaningful gain at this scale.

### S4: Hoist inline regexes in buildChartParts

- **File**: `api/chat.ts:199`
- **Agent**: Efficiency Sentinel
- **Issue**: `buildChartParts` creates inline regexes `/altitude|height/` and `/velocity|speed/` on every invocation. These should be hoisted to module-level constants like the intent detection regexes above.

### S5: Cache buildSystemPrompt with TTL

- **File**: `api/chat.ts:70-108`
- **Agent**: Efficiency Sentinel
- **Issue**: `buildSystemPrompt()` reconstructs the entire multi-KB system prompt string on every text-intent request. The static portion (~3 KB) never changes; the dynamic portion (UTC time, MET, phase) changes at most once per minute. Caching with a 60-second TTL would eliminate redundant string concatenation.

### S6: VALID_ROLES Set overhead for 2 values

- **File**: `api/chat.ts:119`
- **Agent**: Efficiency Sentinel
- **Issue**: `VALID_ROLES` is a `Set` with only 2 entries. For two values, `role === 'user' || role === 'model'` is equivalent or marginally faster. Not a real performance concern — the Set adds clarity about intent (membership check) at the cost of minor overhead.

### S7: toLocaleString slow in V8

- **File**: `api/chat.ts:94`
- **Agent**: Efficiency Sentinel
- **Issue**: `toLocaleString` with `Intl` options is 10-100x slower than manual string formatting in V8. Called on every text-intent request with user timezone. Consider manual formatting or caching the result.
