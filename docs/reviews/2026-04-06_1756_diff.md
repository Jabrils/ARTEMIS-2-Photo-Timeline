## Warnings

### W1: Full message history copied and sent to API on every message

- **File**: `src/hooks/useChat.ts:44`
- **Agent**: DS&A Reviewer, Efficiency Sentinel
- **Issue**: Every `sendMessage` call creates a full copy of the message history with `[...messagesRef.current, userMsg].map(...)`. For long conversations (50+ messages), this creates 2 intermediate arrays and serializes all messages into the fetch body. The server already slices to the last 20 messages (`api/chat.ts:277`), so messages beyond the last 19 are transmitted and immediately discarded.
- **Suggested approach**: Slice `messagesRef.current` to the last 19 messages client-side before spreading + mapping. This reduces serialization, network payload, and GC pressure without changing behavior.

## Suggestions

### S1: IIFE/new URL() allocation per grounding chunk

- **File**: `api/chat.ts:170`
- **Agent**: Complexity Analyst, Efficiency Sentinel
- **Issue**: The fallback title extraction uses an IIFE with `new URL()` inside `.map()`, creating a closure + URL object per grounding chunk. Bounded by API response (typically 1-5 chunks), so impact is negligible. Could use regex hostname extraction instead.

### S2: Repeated Intl.DateTimeFormat instantiation per message

- **File**: `src/hooks/useChat.ts:54`
- **Agent**: Efficiency Sentinel
- **Issue**: `Intl.DateTimeFormat().resolvedOptions().timeZone` is called on every `sendMessage`, instantiating a new DateTimeFormat object each time. The user's timezone doesn't change within a session. Hoist to module-scope constant or useRef.

### S3: useEffect ref sync timing gap

- **File**: `src/hooks/useChat.ts:24`
- **Agent**: Efficiency Sentinel
- **Issue**: `messagesRef` updates asynchronously after render via useEffect. `sendMessage` compensates by manually appending `userMsg`. A synchronous ref update in a custom setState wrapper would eliminate the timing gap. Not a bug at human typing speed.

### S4: Inline style object allocation in TypingIndicator

- **File**: `src/chat/ChatMessage.tsx:77-82`
- **Agent**: Efficiency Sentinel
- **Issue**: TypingIndicator creates 3 new `{animationDelay}` style objects on every render. Since delays are static, hoist to module-scope constants. Only renders while loading — impact negligible.

### S5: Quick answer linear scan at N=12

- **File**: `src/data/artemis-knowledge.ts:63-71`
- **Agent**: DS&A Reviewer
- **Issue**: `findQuickAnswer()` linear scan with `String.includes()` is appropriate at N=12. If FAQ grows to 100+, consider a keyword index. No action needed now.

### S6: Curated video linear scan at N=7

- **File**: `api/chat.ts:25-33`
- **Agent**: DS&A Reviewer
- **Issue**: `CURATED_VIDEOS.find()` + `.some()` is O(n*k) but n=7, k~3. A `Map<keyword, video>` would give O(1) but is over-engineering at this scale. No action needed now.
